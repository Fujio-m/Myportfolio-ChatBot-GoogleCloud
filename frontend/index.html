プロジェクト概要

開発の背景・動機

私は現在、勤怠管理ツールの担当者として、新規入職者からの問い合わせ対応や、ルールの確認・エスカレーション業務に従事しています。

社内にチャットボットが導入されてからは、定型的な問い合わせへの対応時間が月約3時間削減されるという効果を担当者として実感しました。

「自分でも同等のシステムを設計・構築できる力を身につけたい」という動機から、実業務の課題をテーマに本ポートフォリオを開発しました。

チャットボット導入により解決された課題

  • 自己解決率の向上

    新規入職者や、ルールに迷った従業員が、担当者に聞く前に、まずチャットボットに質問して自分で調べられる環境になった

  • 対応工数の削減

    繰り返し発生する定型的な問い合わせをチャットボットで自動化し、担当者の対応する時間が低減された

  • メンテナンス負担の低減

    勤怠管理ルールが変更になっても、管理用PDFを差し替えるだけでAIの回答内容を最新化でき、プログラム改修を不要にする

プロジェクト再構築の背景

再構築の背景

最初はStreamlitで開発しましたが、実務で広く使われている「フロントエンドとバックエンドを分離したWeb構成」への移行を考えて、実務に近い開発手法を経験するとともに、これまでに学習したGoogle Cloudやコンテナ（Docker）のスキルを使用できる成果物に再構築しました。

サービスの選定理由

フロントエンド：Firebase Hosting
  • Google Cloudと親和性が高く、無料枠を使用してWebページを公開できるため
バックエンド：Cloud Run
  • コストの最適化：リクエストがない時はサーバー数を「0」に自動で縮小でき、無料枠を活用して極力コストを0に抑えられるため。
  • スキルの実践：自身が学習してきたCloud技術を直接活かせれるため。

チャットボットアプリの流れ (シーケンス図)

1. 初期化プロセス（安全な起動準備）

アプリが起動した際、セキュリティを考慮し環境変数やSecrets設定ファイルから安全にAPI認証情報を読み込みます。その後、就業規則PDFからテキストデータを抽出し、検索用メモリ空間を確保します。

2. 対話プロセス（ユーザーとの対話フロー）

ユーザーが質問を入力すると、その質問内容とPDFテキストを組み合わせてシステムプロンプトに注入します。Gemini 2.5 Flash-liteモデルに送り、コンテキストに基づいた厳格な回答をUIに表示します。

×

設計におけるこだわり

1. ハルシネーションの抑制

就業規則原本（PDF）から検索した文脈以外の「推測による回答」を禁止するシステムプロンプトを設定しています。答えられない質問には無理に答えず、正しく「回答不可」と判定することで、AIの不正確な回答（ハルシネーション）を防ぎます。

2. 機密情報の保護

Gemini APIキー等の認証情報は、機密情報を安全に一元管理する Google Cloud Secret Manager に格納。ソースコードや設定ファイルへのハードコードを徹底排除し、漏洩リスクを最小化しています。また、PDF原本自体もコンテナ内の安全なメモリ上に読み込む構成としています。

3. 改修不要なメンテフリー設計

就業規則の改訂時にも、管理者が最新のPDFファイルをアップロードして上書きするだけで、AIの回答根拠が即座に最新化されます。プログラムの修正や再デプロイが不要なため、運用の管理コストを劇的に削減します。
