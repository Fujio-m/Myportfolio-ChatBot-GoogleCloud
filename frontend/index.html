プロジェクト概要

開発の背景・動機

私は現在、勤怠管理ツールの担当者として、新規入職者からの問い合わせ対応や、ルールの確認・エスカレーション業務に従事しています。

社内にチャットボットが導入されてからは、定型的な問い合わせへの対応時間が月約3時間削減されるという効果を担当者として実感しました。

「自分でも同等のシステムを設計・構築できる力を身につけたい」という動機から、実業務の課題をテーマに本ポートフォリオを開発しました。

チャットボット導入により解決された課題

  • 自己解決率の向上

    新規入職者や、ルールに迷った従業員が、担当者に聞く前に、まずチャットボットに質問して自分で調べられる環境になった

  • 対応工数の削減

    繰り返し発生する定型的な問い合わせをチャットボットで自動化し、担当者の対応する時間が低減された

  • メンテナンス負担の低減

    勤怠管理ルールが変更になっても、管理用PDFを差し替えるだけでAIの回答内容を最新化でき、プログラム改修を不要にする

プロジェクト再構築の背景

再構築の背景

最初はStreamlitで開発しましたが、実務で広く使われている「フロントエンドとバックエンドを分離したWeb構成」への移行を考えて、実務に近い開発手法を経験するとともに、これまでに学習したGoogle Cloudやコンテナ（Docker）のスキルを使用できる成果物に再構築しました。

サービスの選定理由

フロントエンド：Firebase Hosting
  • Google Cloudと親和性が高く、無料枠を使用してWebページを公開できるため
バックエンド：Cloud Run
  • コストの最適化：リクエストがない時はサーバー数を「0」に自動で縮小でき、無料枠を活用して極力コストを0に抑えられるため。
  • スキルの実践：自身が学習してきたCloud技術を直接活かせれるため。

アプリケーション全体のデータフロー

1. 初期化プロセス（安全な起動準備）

Cloud Run 上のアプリが起動する際、機密情報はソースコードに持たさず、Secret Manager から安全にAPIキーを取得します。

その後、RAG検索用に就業規則PDFからテキストデータを抽出します。

2. 対話プロセス（ユーザーとの対話フロー）

ユーザーが質問を入力すると、その質問内容とメモリ内のPDFテキストを結合し、専用のシステムプロンプトを構築します。

これを Gemini APIに送信することで、社内規定に沿った回答を生成します。

×

設計におけるこだわり

1. ハルシネーション防止の徹底

就業規則PDF以外の「推測による回答」を禁じるシステムプロンプトを設定。
回答できない質問には「回答不可」と明示させることで、AIのハルシネーションの防止を徹底しました。

2. 機密情報の保護

Gemini APIキー等の認証情報は Secret Manager で一元管理し、ソースコードからの情報漏洩を徹底排除しています。
PDF原本もコンテナ内のメモリ上に配置し、外部露出を防ぐ設計にしました。

3. PDF更新によるPDCAサイクル

就業規則の改訂時は管理者がPDFを更新するだけでAIの回答が即座に最新化される設計にしました。
更新作業もあえて人間が行うことで、AIだけに任せた場合のリスクを排除しました。

4. ポートフォリオ全体のコスト運用

Cloud Run・Firebase Hosting・Secret Manager・Artifact Registryの各サービスを無料枠の範囲内で運用できるよう設計。
北米リージョン(us-central1)選択やインスタンスの自動縮小など、コストを意識した構成を徹底しています。
